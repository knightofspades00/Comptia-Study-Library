Why modules exist

A module is a Python file (.py) you can import into another file to reuse its code. Modules let you split a big program into manageable files and use functionality the standard library already provides.

Four import variants — all on the exam

# 1. Import the whole module — access via dotted name
import math
print(math.sqrt(16))      # 4.0

# 2. Import specific names from a module — use directly
from math import sqrt, pi
print(sqrt(16), pi)       # 4.0 3.14159...

# 3. Import everything (avoid — pollutes your namespace)
from math import *
print(sqrt(16))           # works but you cannot tell where sqrt came from

# 4. Import with an alias (common for numpy/pandas)
import math as m
print(m.sqrt(16))         # 4.0
📌 Exam trap — from X import *

Avoid in real code (pollutes the namespace), but expect a question asking what it does. Also: __all__ inside a module defines what * imports — when present, * only pulls the names listed there.

Building a package — __init__.py

A package is a folder containing an __init__.py file (which can be empty) and other .py module files. Importing a package runs its __init__.py.

myapp/
├── __init__.py            # makes myapp a package
├── utils.py
└── network/
    ├── __init__.py
    └── client.py

# In another file:
from myapp.network.client import connect
import myapp.utils as u

PIP — installing third-party packages

  • pip install requests — install the latest version of a package from PyPI (Python Package Index)
  • pip install requests==2.31.0 — pin a specific version
  • pip install -r requirements.txt — install every package listed in a file
  • pip list — list installed packages
  • pip show requests — show metadata + dependencies
  • pip uninstall requests — remove a package
  • pip freeze > requirements.txt — save the current environment to a lock file
💼 Real-world — virtual environments

Never pip install into your system Python in real projects. Create a virtual environment per project: python -m venv .venv then source .venv/bin/activate (or .venv\Scripts\activate on Windows). Now pip install only affects that project. Standard practice everywhere.

📚 W3Schools ▼

Why exceptions matter

Things go wrong: a file does not exist, a user types letters instead of a number, the network drops. Exceptions are Python's structured way to signal and handle these errors without crashing the program.

try / except / else / finally — the full shape

try:
    # code that might raise an exception
    n = int(input("Enter a number: "))
    result = 10 / n
except ValueError:
    print("That was not a number")
except ZeroDivisionError:
    print("Cannot divide by zero")
except Exception as e:
    print(f"Something else broke: {e}")
else:
    # runs ONLY if the try block had no exception
    print(f"Got {result}")
finally:
    # runs no matter what — cleanup goes here
    print("Done")
  • try — code that might raise.
  • except SomeError — catch a specific exception. Multiple except blocks check top-to-bottom.
  • except Exception as e — catch anything that subclasses Exception; bind the instance to e.
  • else — runs only when no exception fired. Optional.
  • finally — runs always (success, exception, even a return). Used for cleanup (close files, release locks).

Exception hierarchy — pick the right one

BaseException
 ├── SystemExit
 ├── KeyboardInterrupt
 └── Exception            ← catch THIS for normal errors
      ├── ArithmeticError
      │    └── ZeroDivisionError
      ├── LookupError
      │    ├── IndexError      # list[100] when len = 3
      │    └── KeyError        # dict["missing"]
      ├── ValueError           # int("abc")
      ├── TypeError            # 1 + "x"
      ├── FileNotFoundError    # open("missing.txt")
      ├── PermissionError
      ├── OSError
      └── many more...
📌 Exam favorite — DO NOT catch BaseException

Catching BaseException (or bare except:) also catches KeyboardInterrupt and SystemExit — meaning Ctrl+C won't stop your program. Always catch Exception at most, or specific subclasses ideally.

Raising your own exceptions

def withdraw(amount):
    if amount < 0:
        raise ValueError("Amount must be positive")
    if amount > balance:
        raise InsufficientFundsError(amount, balance)
    return balance - amount

# Custom exception class — inherit from Exception
class InsufficientFundsError(Exception):
    def __init__(self, requested, available):
        self.requested = requested
        self.available = available
        super().__init__(f"Need {requested}, have {available}")

Chained exceptions — preserve the original cause

When you catch one exception and raise another, use raise X from Y to keep the original visible in the traceback. Otherwise the original gets lost.

try:
    config = json.loads(text)
except json.JSONDecodeError as e:
    raise ConfigError("Invalid config file") from e
# Traceback now shows BOTH the JSONDecodeError and the ConfigError

Assertions — quick sanity checks

assert raises AssertionError if the expression is falsy. Used for "this should never happen" checks in development. Disabled when Python runs with -O (optimize) — don't use assertions for user input validation.

def divide(a, b):
    assert b != 0, "Denominator must not be zero"
    return a / b
💼 Real-world rule — be specific, be early, fail loudly

Catch the most specific exception you can handle. Avoid silently catching everything — that hides bugs. If you can't fully recover, let the exception propagate up or re-raise it.

Dicts — key/value lookup

List comprehensions — the Python way to build lists

# Verbose

for x in range(10):
    squares.append(x * x)

# Comprehension — same result, one line

Lambda — single-expression anonymous functions

# A function in one line
double = lambda x: x * 2
double(5)    # 10

# Most useful as a key= argument
words = ["banana", "apple", "cherry"]
words.sort(key=lambda w: len(w))           # sort by length

📌 Mutability rule

Immutable types: str, int, float, bool, tuple, frozenset, bytes. Mutable: list, dict, set, bytearray. Only immutable types can be dict keys or set members.

Classes and instances — the basics

class Dog:
    species = "Canis familiaris"      # class variable (shared by ALL dogs)

    def __init__(self, name, age):     # constructor — called by Dog(...)
        self.name = name               # instance variables (per-dog)
        self.age = age

    def bark(self):                    # instance method
        return f"{self.name} says woof!"

# Create instances
rex = Dog("Rex", 3)
fido = Dog("Fido", 5)

rex.bark()         # 'Rex says woof!'
rex.species        # 'Canis familiaris' — read from class
Dog.species        # 'Canis familiaris' — same thing via class
rex.age = 4        # mutate an instance variable
  • class Dog: defines a new type. PEP 8 convention: PascalCase for class names.
  • __init__ is the constructor. Python calls it when you do Dog(...).
  • self is the instance the method is being called on. Always the first parameter of instance methods. The name self is convention, not enforced — but always use it.
  • Class variables are shared across all instances. Instance variables are per-instance (set via self.x = ...).

Three flavors of methods

class Counter:
    count = 0      # class variable

    def __init__(self, name):
        self.name = name
        Counter.count += 1

    def describe(self):                    # INSTANCE method — needs self
        return f"{self.name} (one of {Counter.count})"

    @classmethod
    def total(cls):                        # CLASS method — receives cls (the class)
        return cls.count

    @staticmethod
    def greet():                           # STATIC method — no self, no cls
        return "hello from Counter"

a = Counter("a")
b = Counter("b")
a.describe()        # 'a (one of 2)'
Counter.total()     # 2
Counter.greet()     # 'hello from Counter'

Encapsulation — public, _protected, __private

Python doesn't have true private. Conventions:

  • name — public
  • _name — "protected"; convention says "internal, don't touch from outside"
  • __name — name-mangled to _ClassName__name so it doesn't collide with subclasses; still technically accessible
class Account:
    def __init__(self):
        self.owner = "Alice"      # public
        self._note = "internal"   # protected
        self.__pin = 1234         # name-mangled

a = Account()
a.owner          # 'Alice'         — fine
a._note          # 'internal'      — works but you "shouldn't"
a.__pin          # AttributeError  — name-mangled
a._Account__pin  # 1234            — escape hatch

Inheritance — reuse and specialize

class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "(silence)"

class Dog(Animal):               # Dog inherits from Animal
    def speak(self):             # override
        return f"{self.name} says woof!"

class GuideDog(Dog):
    def __init__(self, name, owner):
        super().__init__(name)   # call Animal.__init__
        self.owner = owner

g = GuideDog("Luna", "Sam")
g.speak()             # 'Luna says woof!'  (uses Dog's override)
isinstance(g, Dog)    # True
isinstance(g, Animal) # True
issubclass(GuideDog, Animal)  # True
  • class Sub(Super): declares inheritance. Sub gets all of Super's methods + attributes.
  • super() — call a parent class's method. Most often used inside __init__ to set up the parent.
  • Method overriding — child defines a method with the same name; child's version wins for instances of the child.
  • isinstance(obj, Cls) — true if obj is an instance of Cls or any subclass.

Polymorphism — same method, different behavior

class Cat(Animal):
    def speak(self):
        return f"{self.name} says meow"

animals = [Dog("Rex"), Cat("Whiskers"), Dog("Buddy")]
for a in animals:
    print(a.speak())     # each runs its own speak() — that's polymorphism

Multiple inheritance and MRO

Python allows a class to inherit from multiple parents. The Method Resolution Order (MRO) decides which parent's method wins — Python uses the C3 linearization algorithm.

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

D.__mro__   # (D, B, C, A, object)
# Method lookup walks this order, returning the first match

@property — getters and setters without ceremony

class Temperature:
    def __init__(self, celsius):
        self._celsius = celsius

    @property
    def celsius(self):           # read like an attribute: t.celsius
        return self._celsius

    @celsius.setter
    def celsius(self, value):    # write like an attribute: t.celsius = 25
        if value < -273.15:
            raise ValueError("Below absolute zero")
        self._celsius = value

    @property
    def fahrenheit(self):        # computed property — no setter
        return self._celsius * 9/5 + 32

t = Temperature(100)
t.celsius         # 100
t.fahrenheit      # 212.0
t.celsius = 25
t.fahrenheit      # 77.0

Iterators — the protocol behind for

An iterable is anything you can loop over (lists, strings, dicts, files, generators). An iterator is the object that tracks position. iter(x) turns an iterable into an iterator; next(it) pulls the next value.

nums = [10, 20, 30]
it = iter(nums)           # <list_iterator object>
next(it)                  # 10
next(it)                  # 20
next(it)                  # 30
next(it)                  # StopIteration — raised, not returned

# A for loop is just sugar for this:
for n in nums:            # 1. it = iter(nums)
    print(n)              # 2. while: try n = next(it); except StopIteration: break

Custom iterator — implement __iter__ + __next__

class Countdown:
    def __init__(self, start):
        self.n = start

    def __iter__(self):
        return self                    # the iterator IS the object

    def __next__(self):
        if self.n <= 0:
            raise StopIteration
        current = self.n
        self.n -= 1
        return current

for x in Countdown(3):    # 3, 2, 1
    print(x)

Generators — iterators written as functions

A function with yield is a generator. Each yield pauses execution and returns the value. Next call resumes after the yield. Lazy — only computes what's asked for.

def countdown(n):
    while n > 0:
        yield n
        n -= 1

for x in countdown(3):    # 3, 2, 1
    print(x)

# Equivalent to the Countdown class above — but in 4 lines, not 11

# Generator expression (parentheses instead of brackets)
squares = (x*x for x in range(1_000_000))     # no list in memory
total = sum(x*x for x in range(1_000_000))    # streams through 1M values
                                              # with constant memory

Closures — functions that "remember" their enclosing scope

def make_counter():
    count = 0                          # local to make_counter
    def increment():
        nonlocal count                 # write to the enclosing scope
        count += 1
        return count
    return increment                   # return the inner function

c = make_counter()
c()    # 1
c()    # 2
c()    # 3
# count survives between calls because increment "closes over" it

nonlocal writes to the enclosing function's scope; global writes to the module-level scope. Without one of those, an assignment creates a new local variable.

File I/O — text and binary modes

# READING — old style (must remember to close)
f = open("data.txt", "r")
contents = f.read()
f.close()

# READING — modern with statement (auto-closes, even on exceptions)
with open("data.txt", "r") as f:
    contents = f.read()         # whole file as string
    # OR
    for line in f:              # line-by-line — best for big files
        print(line.rstrip())

# WRITING
with open("output.txt", "w") as f:    # "w" truncates the file
    f.write("Hello\n")
    f.write("World\n")

with open("log.txt", "a") as f:       # "a" appends
    f.write("Another line\n")

💼 Why with?

The with statement is a context manager — it guarantees cleanup runs even if an exception fires. Always use with open(...) for files. Same pattern for database connections, network sockets, locks. Implementation: a class with __enter__ and __exit__ methods.

🔒 Mutable vs Immutable — Know the Difference

This is the single most-tested concept on PCAP. Get it wrong and a third of OOP and data-structure questions slip away.

The aliasing trap
y = x doesn't copy a mutable object — it makes y another name for the SAME object. y.append(...) changes what x sees too. To actually copy: y = x.copy() or y = list(x) or y = x[:]. For nested structures use copy.deepcopy(x).

📋List vs tuple — when do I use which?
Scenario: You need to store coordinates, configuration values, names of weekdays, a shopping list, and a set of (user_id, action) pairs. List or tuple — and why?
Step 1 — The single distinction

list is mutable; you change it after creation. tuple is immutable; once created, you cannot change it.

Everything else follows from that one fact.

Step 2 — The decision tree
If your data...UseWhy
Will change (add, remove, sort)listMutability is required
Is a "record" — fixed structuretupleSignals "this won't change"; faster, smaller
Needs to be a dict keytupleImmutable = hashable; lists are not hashable
Needs to be a set membertupleSame — sets need hashable items
Function returning multiple valuestupleConvention; supports unpacking
You don't know yetlistMore common default, more flexible
Step 3 — Apply to the scenario
  • Coordinates like (x, y): tuple — fixed length, won't mutate
  • Configuration values (debug=True, port=8080): tuple (or a named tuple / dataclass)
  • Names of weekdays: tuple — never changes
  • Shopping list: list — you add and remove items
  • (user_id, action) pairs as set members: tuple — sets need hashable items
Exam shortcut

If the question mentions "used as a dict key" or "member of a set" or "returning multiple values" — answer is almost always tuple. If it mentions "appending", "sorting in place", or "changing" — list.

🏛️Build a class step by step — Dog → Animal → GuideDog
Scenario: Sketch a small class hierarchy that exercises every OOP pillar tested on PCAP — constructors, instance vs class variables, instance methods, inheritance, super(), method overriding, magic methods.
Step 1 — Start with the base class 
class Animal:
    kingdom = "Animalia"                   # class variable

    def __init__(self, name, age=0):
        self.name = name                   # instance variables
        self.age = age

    def speak(self):                       # default behavior
        return f"{self.name} makes a noise"

    def __str__(self):                     # magic method
        return f"{type(self).__name__}({self.name!r})"
  • kingdom is a CLASS variable — every Animal shares it.
  • self.name and self.age are INSTANCE variables — each Animal has its own.
  • __str__ controls what print(animal) shows.
Step 2 — Subclass that overrides 
class Dog(Animal):
    def __init__(self, name, age, breed):
        super().__init__(name, age)        # call Animal's __init__
        self.breed = breed                 # ADD instance var

    def speak(self):                       # OVERRIDE
        return f"{self.name} says woof!"
  • super().__init__(name, age) hands the name + age to Animal so its setup runs.
  • Then Dog adds its OWN attribute (breed).
  • Dog.speak overrides Animal.speak. Dogs say woof; generic animals make noise.
Step 3 — One more level: specialize Dog into GuideDog 
class GuideDog(Dog):
    def __init__(self, name, age, breed, owner):
        super().__init__(name, age, breed) # call Dog's __init__
        self.owner = owner

    def speak(self):                       # override again
        return f"{self.name} guides {self.owner}"

g = GuideDog("Luna", 5, "Lab", "Sam")
print(g)            # GuideDog('Luna')
print(g.speak())    # Luna guides Sam
print(g.kingdom)    # Animalia  ← inherited all the way up
isinstance(g, Animal)   # True
isinstance(g, Dog)      # True
Pillars exercised
  • Encapsulation — name and age live inside the object.
  • Inheritance — Dog inherits from Animal, GuideDog from Dog. Each adds attributes.
  • Polymorphism — three classes, same speak() name, three behaviors.
  • super() — used in both subclasses to call the parent's __init__.
  • Magic methods — __init__ for setup, __str__ for printing.

If you can write this from scratch in 5 minutes, you're ready for the OOP section.

📜Read this traceback — what failed and why?
Scenario: Your code crashes with this traceback. Find the bug in under 30 seconds.
Traceback (most recent call last):
  File "app.py", line 18, in <module>
    print(top_buyer(orders))
  File "app.py", line 12, in top_buyer
    return max(totals, key=totals.get)
  File "app.py", line 8, in build_totals
    totals[o["user"]] += o["amount"]
KeyError: 'amount'
Step 1 — Read top-to-bottom: "most recent call last"

The bottom-most frame is where the error fired. The frames above it are the call chain that LED there. Always start at the bottom.

Step 2 — Identify the exception class

KeyError: 'amount' — a dict was accessed with a key that doesn't exist. The key tried was "amount".

Step 3 — Look at the LINE the exception fires on

Bottom frame: app.py line 8, function build_totals, code totals[o["user"]] += o["amount"].

Two dict lookups on that line — but the error said 'amount'. So o["user"] worked; o["amount"] failed. One of the order dicts (o) is missing the "amount" key.

Step 4 — Two reasonable fixes 
# Fix A: skip orders without amounts
for o in orders:
    if "amount" not in o:
        continue
    totals[o["user"]] += o["amount"]

# Fix B: default to 0 when missing
for o in orders:
    totals[o["user"]] += o.get("amount", 0)

# Fix C: fix the data upstream — log + raise a clearer error
for o in orders:
    if "amount" not in o:
        raise ValueError(f"Order missing amount: {o!r}")
    totals[o["user"]] += o["amount"]

Which fix is right depends on whether missing amounts are a bug (raise) or a known data condition (skip / default).

How to read any traceback fast
  1. Read the LAST line — that's the exception class + the message
  2. Read the LAST frame — that's the line that fired
  3. Scan UP the frames — that's the call chain (top of the stack at the bottom)
  4. For library code in the traceback, only worry about it if you suspect you're calling the library wrong

🧠 Tips, Habits & Careers

Habits that make Python click + where the cert leads.

🎯 Habits that make Python click

  • Live in the REPL. Don't just READ about list.append — open Python, type it, see what comes out. Half the exam questions are "what does this print?" — exactly what the REPL tells you.
  • Use a virtual environment every time. python -m venv .venv in every project. Never pip install into system Python.
  • Type the code, don't copy it. Muscle memory for indentation, colons, parentheses, brackets, quotes. Copy-paste teaches you nothing.
  • Read tracebacks bottom-up. The last line is the exception class. The frame just above it is where it fired. Walk up to see the chain.
  • Lean on the docs. help(list) in the REPL or docs.python.org — both are written for beginners and are remarkably clear.
  • Name things obviously. PEP 8: snake_case for variables and functions, PascalCase for classes. n and x are fine for one-line loops; everything else gets a real name.
  • Format with f-strings, not %% or .format(). The exam still tests the old ones; you should recognize them. But write modern code.
  • Test small. Open the REPL, paste a 3-line snippet, run it. Faster than guessing.

Python is the language with the lowest "reading floor" — its syntax reads almost like English, so beginners can ship working code in days.
